  Quantized Training Algorithm:

  1. Initialize FP32 Master Weights
     - Store high-precision weights for gradient accumulation
     - Size: Same as normal transformer

  2. Training Loop (Each Epoch):

     a. Quantize Weights (FP32 → Q8_0)
        - Convert master weights to Q8_0 for inference
        - Memory: 4x reduction (32 bits → 8 bits)

     b. Forward Pass (Q8_0)
        - Use quantized weights for inference
        - Activations in FP32 (for gradient accuracy)
        - Memory savings: 4x

     c. Backward Pass (FP32)
        - Compute gradients in FP32 (critical for accuracy)
        - No memory savings here (need precision)

     d. Update Master Weights (FP32)
        - Apply gradients to FP32 master weights
        - Standard optimizer (Adam, SGD, etc.)

  3. Save Quantized Model
     - Final output: Quantized model (4x smaller)
     - Can be used directly for inference

┌──────────────────────────────────────────────────────────────────────────┐
│ BENEFITS OF QUANTIZED TRAINING                                          │
└──────────────────────────────────────────────────────────────────────────┘

  Memory Savings:
    FP32:  4 bytes/param  (standard)
    Q8_0:  1 byte/param   (quantized)
    Ratio: 4x reduction ✅

  Example - 1B Parameter Model:
    ┌────────────────┬──────────────┬─────────────────┐
    │ Training Mode  │ Memory       │ Can Fit On      │
    ├────────────────┼──────────────┼─────────────────┤
    │ FP32 Standard  │ 4 GB         │ High-end GPU    │
    │ Q8_0 Quantized │ 1 GB         │ Mid-range GPU ✅│
    └────────────────┴──────────────┴─────────────────┘

  Quality Trade-off:
    - Q8_0: < 5% perplexity degradation (acceptable)
    - Q4_0: ~10% degradation (use with caution)
    - Recommendation: Use Q8_0 for training

  Speed Trade-off:
    - Overhead: ~10% slower (quantize/dequantize operations)
    - Net benefit: Can train larger models = worth it!
