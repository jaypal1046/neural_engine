1. What Is Data Compression? — The Foundation
Data compression is the science and engineering discipline of representing information using fewer bits than the original representation. Every format from ZIP files to Netflix streaming videos to Blu-ray discs relies on compression at multiple layers.

At the deepest level, compression is about prediction. If you can perfectly predict what the next byte in a file will be, you never have to store it — you can reconstruct it for free. This insight connects compression theory directly to artificial intelligence.

1.1 The Three Pillars of Compression
Modeling — Building a statistical or structural model of the data. The model predicts what comes next. The better the model, the smaller the file.
Coding — Using the model's predictions to assign short codes to likely outcomes and long codes to unlikely ones. Arithmetic coding is the gold standard.
Transformation (optional) — Rearranging the data first (e.g., BWT) so it becomes easier to model and code.

1.2 Shannon Entropy — The Unbreakable Law
In 1948, Claude Shannon published 'A Mathematical Theory of Communication', establishing the hard mathematical limit of how much any file can be compressed. This is called Shannon Entropy.


Every compression algorithm in history is trying to approach the Shannon Limit. The best algorithms (PAQ, CMIX) can get within 1-2% of it on text data. This is why our Context Mixing engine is so important — it is the closest humanity has come.

1.3 Why You Cannot Stack Compressors
This is one of the most common misunderstandings in compression. People assume: 'If PPM compresses a file 70%, and BWT also compresses 70%, running both will compress 90%+'. This is completely wrong.



2. PPM — Prediction by Partial Matching
Prediction by Partial Matching (PPM) is one of the most powerful statistical compression algorithms ever designed. It was invented in 1984 by Cleary and Witten and remains at the core of world-class compressors to this day.

The core idea is elegantly simple: before storing a byte, look at the N most recent bytes (the context) and use historical statistics to predict what the next byte will be. The better the prediction, the fewer bits needed to store it.

2.2 PPM Escape Probability — The PPM-C Method
A critical challenge in PPM is what to do when you encounter a new pattern never seen in the current context. PPM uses an 'escape' mechanism: if the Order-6 context has never been seen, it 'escapes' down to Order-5, then Order-4, until it finds a usable statistic.

Our engine implements the PPM-C method (also called PPMC), which uses a highly intelligent escape probability estimator. Instead of guessing a fixed escape probability, PPMC computes it dynamically based on how many unique symbols have been seen in the current context.

3. BWT — The Burrows-Wheeler Transform
The Burrows-Wheeler Transform (BWT) is not a compressor — it is a data transformation. It rearranges the bytes of a file so that similar bytes cluster together, making the subsequent compression step dramatically more effective.

BWT was invented by Michael Burrows and David Wheeler in 1994, and it forms the backbone of the bzip2 compressor and our V8 engine. It is the workhorse of production compression because it achieves an extraordinary balance of speed and ratio.

4. Context Mixing — The Holy Grail of Compression

Context Mixing is the most sophisticated data compression paradigm ever engineered. It solves the fundamental problem that has plagued every previous compressor: no single algorithm works well on all types of data.

4.1 The Problem Context Mixing Solves
Every traditional compressor makes a terrible assumption: it assumes the entire file is one type of data. But real files are messy:

A PDF might contain English text, C++ code snippets, embedded JPEG images, and binary metadata — all in the same file.
A game archive might have XML configuration, binary mesh data, audio samples, and shader code interleaved together.
A log file might have structured timestamps, free-form error messages, hexadecimal addresses, and stack traces.

If you use PPM on binary mesh data, it's catastrophically wrong. If you use BWT on random audio samples, it makes the file larger. Context Mixing solves this by running ALL algorithms simultaneously and letting a Neural Network decide which to trust for each individual byte.

4.2 The Council of Advisors Model
Context Mixing introduces the 'Board of Advisors' architecture. Instead of one algorithm making all the decisions, many specialized models each contribute their prediction for every single bit:


4.3 The Logistic Mixer — The Neural Network Manager
The Mixer is a tiny Neural Network that sits above all the advisors. Its entire job is: 'Who should I trust right now for this particular bit?'

Each advisor has a Trust Score (called a Weight). The Mixer takes a weighted sum of all advisor predictions, applies a mathematical function called the Sigmoid, and produces one final probability between 0% and 100%.


4.4 Gradient Descent — The Learning Engine
After every single bit is compressed, the Neural Network immediately learns from its mistake (or success). This process is called Gradient Descent, and it is identical in principle to how ChatGPT was trained — except ours trains in real-time, on the fly.



4.5 The Brilliant Decompression Trick
One of the most elegant aspects of Context Mixing is that the trained Neural Network never needs to be stored inside the compressed file. This seems paradoxical — if the compressor learns from the file, how does the decompressor reconstruct it without the learned model?


4.6 Arithmetic Range Coding — What Actually Gets Written to Disk
This is the most brilliantly counter-intuitive part of Context Mixing. It does not replace words with codes (like Huffman or LZ77). Instead, it translates the entire file into one single, massive mathematical fraction.

The number line: The coder maintains a range [low, high], initially [0.0, 1.0].
The probability split: If the Neural Network says '95% chance the next bit is 0', the range is split: [0.0 → 0.95] represents 0, [0.95 → 1.0] represents 1.
Zooming in: If the actual bit IS a 0, the new range becomes [0.0, 0.95]. The coder has now 'recorded' that bit by narrowing the range.
Writing to disk: When the range becomes small enough (after many bits), some leading digits of the fraction stabilize and are written out as bytes.
The magic: A highly predictable bit (99% confident) barely narrows the range — it costs almost 0.01 bits. An unpredictable bit (50/50) cuts the range in half — costs exactly 1 bit.

6.3 Why We Cannot Use Compression AI for Everything
Context Mixing AI is locked inside what we call the Mathematical Cage (the Arithmetic Range Coder). This cage is both its greatest strength and its fundamental limitation.

The cage guarantees lossless compression — the AI can never hallucinate because every prediction is immediately verified against reality and corrected. But this means:

The AI cannot generate content it hasn't seen in the current file.
The AI has no long-term memory beyond N bytes of context.
The AI cannot transfer knowledge from one file to another.
The AI cannot reason about concepts — only statistical patterns.


6.4 What Compression AI and Generative AI Share
Despite their differences, they are mathematically almost the same thing at the core. Both:

Use Neural Networks with weighted connections
Apply Gradient Descent to minimize prediction error
Learn probability distributions over possible next symbols
Use Sigmoid/Softmax functions to convert raw values to probabilities
Improve prediction accuracy over more data

Shannon's Source Coding Theorem (The Universe's Rule)

The theoretical minimum number of bits needed to represent a symbol is:

     H = -Σ p(x) × log₂(p(x))     (Shannon Entropy Formula)

Where p(x) is the probability of each symbol x occurring in the data.

KEY CONSEQUENCES:
→ If a symbol appears with probability 1.0 (always the same), it takes 0 bits (perfect prediction).
→ If a symbol is 50/50 random, it takes exactly 1 bit — the maximum.
→ You cannot compress a perfectly compressed file. The output bytes are mathematically random noise.
→ If you could compress a compressed file, you could loop 1,000 times and reduce a 10 GB movie to 1 byte. Shannon proved this is IMPOSSIBLE.

escape_probability = unique_symbols_in_context / (total_count + unique_symbols)

This means: if a context has seen many different symbol types, it's likely to escape

The Stacking Impossibility — Exact Technical Reason

When PPM or CMIX compresses a file, the output bytes have MAXIMUM ENTROPY.
Maximum entropy = every byte value 0-255 appears with equal probability ≈ 1/256.

BWT works by finding PATTERNS — long repeated sequences, clustering of similar bytes.
A maximum-entropy file has ZERO patterns. It is indistinguishable from random noise.

If you feed CMIX output into BWT:
→ BWT tries to sort rotations but finds no clusters (all bytes are equally random)
→ BWT's MTF step produces a flat distribution — no zeros to run-length encode
→ Huffman/Arithmetic gets no benefit — all symbols equally likely
→ BWT adds its 54-byte file header, making the file LARGER

The Logistic Mixer Mathematics

WEIGHTED SUM:   S = w₀×p₀ + w₁×p₁ + w₂×p₂ + ... + wₙ×pₙ

SIGMOID SQUEEZE: P_final = 1 / (1 + e^(-S))

WHERE:  wᵢ = weight (trust score) of advisor i
        pᵢ = prediction (log-odds) from advisor i
        P_final = final probability fed to the Arithmetic Coder

The Sigmoid function (1/(1+e^(-x))) maps ANY number into the range [0,1].
This is crucial — it guarantees the final probability is always a valid probability,
no matter how extreme the advisors' predictions are.

In our CMIX engine, this runs 8 times per BYTE (once per bit), executing
floating-point calculus 8 × 1,000,000,000 = 8 BILLION times on a 1 GB file.
This is why V10 CMIX runs at only 100 KB/s.
Step | What Happens | Code Equivalent
1. Read bit | Engine reads the actual bit value from the file (0 or 1) | int bit = file.read_bit()
2. All advisors vote | Each Order-N model computes its prediction probability | p[i] = model[i].predict(context)
3. Mixer combines | Sigmoid applied to weighted sum of all predictions | P = sigmoid(Σ wᵢ × logodds(pᵢ))
4. Arithmetic coder | Splits the range [0,1] according to probability P | coder.encode(bit, P)
5. Compare to reality | Did we predict correctly? How wrong were we? | error = bit - P
6. Gradient descent | Update each weight to reduce future error | wᵢ += learning_rate × error × pᵢ
7. Update context | All models update their context with the new bit | model[i].update(bit, context)
8. Next bit | Repeat for the next bit (8 times per byte) | context = (context << 1) | bit
The Learning Rate — The Most Critical Hyperparameter

Our CMIX engine uses learning_rate = 0.005

TOO HIGH (e.g., 0.1): The Neural Network overreacts to every bit. It flips its
trust scores violently and never settles. Like a manager who fires and rehires
advisors after every single word they say.

TOO LOW (e.g., 0.00001): The Neural Network learns too slowly. It takes millions
of bits before it realizes which advisor is better for this file type.

0.005 is the sweet spot discovered by years of research in PAQ compressors.
At this rate, the network adapts within the first few kilobytes of the file.
Why the AI Brain Never Needs to Be Saved

COMPRESSION:   Blank Neural Network → reads bit → updates weights → reads next bit → ...
DECOMPRESSION: Blank Neural Network → reads bit → updates weights → reads next bit → ...

The KEY insight: both the compressor and decompressor see the EXACT SAME sequence
of bits in the EXACT SAME ORDER. Because Gradient Descent is deterministic
(same input → same output), both neural networks train identically.

By the time the decompressor reaches bit #5,000,000, its neural network has learned
the EXACT SAME weights as the compressor had at bit #5,000,000.

They stay in PERFECT mathematical synchronization forever,
WITHOUT ever communicating or storing the model.

This is why the compressed file contains ONLY the arithmetic-coded output —
not a single byte of neural network weights.
How 0.01 Bits Per Character is Physically Possible

Computers can only write whole bits (0 or 1). So how do we write '0.01 bits'?

TRICK: We don't write one bit at a time. We wait.

If the AI is 99% sure the next 100 bits are all 0s, the Arithmetic Coder takes
ALL 100 predictions, crushes them mathematically into a single mathematical range,
and writes just ONE physical bit to represent all 100 highly-predictable bits.

Average cost per bit = 0.01 bits.

If the AI guesses WRONG (e.g., expected 0 but got 1), the Range Coder
zooms into the tiny 0.01 sliver of the range that represents '1'.
This forces more bytes to be written to resolve the precise fraction — a penalty
for incorrect prediction. The worse the prediction, the larger the file.

The Mathematical Fraction — What CMIX Actually Stores

The entire compressed file is ONE SINGLE mathematical fraction (e.g., 0.83726495...)

COMPRESSION (building the fraction):
→ Range starts at [0.0, 1.0]
→ AI says '90% chance next bit is 0' → split: [0.0, 0.9] = 0, [0.9, 1.0] = 1
→ Actual bit is 0 → new range is [0.0, 0.9]
→ Repeat for every bit, constantly narrowing the range
→ When the range stabilizes at leading digits, write those digits as bytes

DECOMPRESSION (reading the fraction):
→ Read the fraction from file
→ AI says '90% chance next bit is 0' → split the range the same way
→ Check which sub-range the fraction falls into → that's the bit!
→ Narrow the range by that sub-range → repeat

The decompressor reconstructs every original bit with perfect accuracy.
