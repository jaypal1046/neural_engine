# Format-Specific Writing Guidance

## Blog Post / Article

**Structure**: Hook → Problem → Solution/Insight → Evidence → Conclusion
**Length**: 800–2500 words typical. Under 800 = too thin. Over 3000 = needs subheadings every 400 words.
**Hook types that work**: Personal story, surprising stat, counter-intuitive claim, direct question
**Common mistakes**: Burying the main point past paragraph 3, weak transitions, ending without a takeaway

---

## Newsletter

**Structure**: Hook (1–2 lines) → Main insight (200–400 words) → 1 supporting example → Takeaway/CTA
**Tone**: More personal and direct than blog posts. Write like you're emailing one person.
**Length**: 300–600 words for weekly. 600–1200 for deep-dive editions.
**Common mistakes**: Too many topics in one edition, forgetting the "so what", burying the CTA

---

## Tutorial / How-To

**Structure**: What you'll learn → Prerequisites → Steps (numbered) → Expected result → Troubleshooting
**Clarity rules**:
- One action per step
- Show expected output after each step
- Distinguish commands from explanations visually (code blocks)
- Test every instruction before publishing

**Common mistakes**: Assuming knowledge the reader doesn't have, skipping prerequisites, no troubleshooting section

---

## Thought Leadership

**Structure**: Provocative claim → Why conventional wisdom is wrong → Evidence for your view → Implications → Call to action
**Requirements**: Must have a genuine POV — not "here are both sides." The writer must take a stand.
**Research need**: At least 2–3 credible sources, plus 1 real example (ideally from the writer's own experience)
**Tone**: Confident but not arrogant. Back up strong claims.
**Common mistakes**: Being too safe, hedging the main argument, forgetting to say what the reader should *do* with this insight

---

## Case Study

**Structure**: Context (company/situation) → Problem → What they tried → What worked → Results (with numbers) → Lessons
**Requirements**:
- Specific metrics whenever possible ("saved 3 hours/week" not "saved time")
- Quote from someone involved if possible
- Honest about what didn't work — credibility comes from nuance

**Common mistakes**: Only showing wins, vague outcomes, no transferable lesson for the reader

---

## Technical Documentation

**Structure**: Overview → Prerequisites → Concepts (brief) → Steps → Reference → Troubleshooting
**Rules**:
- Version-stamp everything (software versions, dates)
- Code examples must actually run
- Use consistent terminology — pick one word for each concept
- Link to related docs at the end

**Common mistakes**: Outdated examples, inconsistent naming, no error handling guidance
